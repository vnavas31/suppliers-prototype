# Simplifae Ingestion Pipeline

Prototype pipeline for turning a PLACSP tender deeplink into a source-backed `premium.json` packet before touching the UI prototype.

## Method Contracts

Read these files before changing the ingestion method or promotion logic:

- `SIMPLIFAE_METHOD.md`: fast/slow lane rules and premium evidence contract.
- `PLACSP_SOURCE_MAP.md`: source priority and field-by-field source policy.
- `OCR_POLICY.md`: OCR downgrade and future OCR lane policy.
- `VIEWER_NAVIGATION_CONTRACT.md`: Tender Detail/source-chip/document-viewer contract.
- `.claude/`: reusable agent commands, rules, and hook guidance for team workflows.

## Command Shortcuts

Use the wrapper for reproducible local/CI runs:

```bash
node tools/simplifae-ingestion/simplifae.js check
node tools/simplifae-ingestion/simplifae.js smoke --premium-root /path/to/premium
```

The wrapper keeps safe commands non-mutating by default. Promotion into `discovery-v2` remains an intentional manual step because it changes prototype files.

## Output

- `raw/detail.html` and `raw/detail.txt`: fetched PLACSP detail page.
- `documents/`: all linked documents plus XML-embedded PCAP/PPT/MJ references.
- `text/`: page-level text for each processed PDF.
- `premium.json`: structured facts, downloaded document metadata, candidate evidence pages by section, coverage metrics, and a first `level3` auto-draft for Tender Detail fields.
- `packet-manifest.json`: batch-level canonical manifest generated by the quality loop. It validates local document integrity, source metadata, document/page evidence resolvability, OCR downgrade status, and which packets are eligible for promotion.

The `level3` block currently includes:

- `summary`: source-backed scope summary.
- `timeline`: publication, deadline and submission mode from structured PLACSP data.
- `commercial_facts`: estimated value, base budget, duration and guarantees.
- `scope`: lots, CPVs, lot budgets and source evidence.
- `requirements`: envelope-aware admission requirements and award criteria.

The pipeline handles both direct PLACSP document links and aggregated `contractaciopublica.cat` document links surfaced through PLACSP detail pages.

It also expands ZIP packages when they contain PDF/XML evidence files. This matters for profiles that publish PCAP, PPT, annexes or characteristic tables as compressed packages instead of direct PDFs.

## Training Batch

```bash
python3 tools/simplifae-ingestion/train_batch.py \
  --limit 50 \
  --out /tmp/simplifae-ingestion/training-publicada
```

The training runner pulls `PUB` tenders from the official PLACSP Atom feed, runs the pipeline with a per-tender timeout, and writes `training-report.json` plus newline-delimited results for later analysis.

The Atom feed is useful for backfill, regression training and daily reconciliation, but it is not the production hot path because PLACSP can publish new tenders before the feed refreshes.

For the production hot path, use the structured fast path first. This mode fetches the live tender detail page, extracts structured Discovery fields, and intentionally skips document downloads/PDF parsing so new tenders can appear quickly:

```bash
python3 tools/simplifae-ingestion/train_batch.py \
  --limit 50 \
  --seed-file /tmp/simplifae-ingestion/live-publicada-50.json \
  --no-atom \
  --workers 12 \
  --pipeline-mode discovery \
  --out /tmp/simplifae-ingestion/discovery-live-publicada-50
```

Then queue selected/high-value tenders for premium extraction. This mode downloads documents, expands XML/ZIP references, extracts page-level text, and builds source-backed evidence:

```bash
python3 tools/simplifae-ingestion/train_batch.py \
  --limit 50 \
  --seed-file /tmp/simplifae-ingestion/live-publicada-50.json \
  --no-atom \
  --workers 6 \
  --pipeline-mode premium \
  --out /tmp/simplifae-ingestion/premium-live-publicada-50
```

OCR is a separate enrichment lane for premium tenders still blocked by scanned/low-text PDFs:

```bash
node tools/simplifae-ingestion/simplifae.js ocr-50 \
  --workers 6 \
  --timeout 420 \
  --ocr-timeout 240
```

The OCR lane keeps the fast path untouched. It runs local OCR only on documents still classified as `ocr_candidate`, stores OCR-derived PDFs under each packet `ocr/` directory, re-extracts page text, and re-runs the same quality/semantic/parity/manifest gates.

Production ingestion should use live search for near-real-time discovery, Atom for reconciliation, and a separate historical awards/adjudications dataset for competitors, partner match and success-rate signals.

The training report includes operational readiness metrics:

- `ready_for_discovery`: enough structured/summary/deadline data to show a card quickly.
- `premium_ready`: high-confidence premium packet with no missing evidence blocks.
- `needs_ocr`: PDFs downloaded but no useful section evidence extracted.
- `needs_documents`: document acquisition still failed.
- `legal_refinement_only`: remaining gaps are guarantees or required-document criteria.

Guarantee extraction is conservative: explicit `not required`, percentage, amount, or document-page evidence is preferred. When provisional guarantee is not explicitly required but the guarantee section is reviewed, the pipeline records `Not stated as required in reviewed guarantee evidence` instead of inventing a stronger legal conclusion.

Each result also includes an `operational_status`:

- `premium_ready`
- `discovery_ready_structured`
- `discovery_ready_needs_ocr`
- `discovery_ready_needs_legal_review`
- `discovery_ready_needs_confidence_review`
- `needs_ocr`
- `needs_documents`
- `needs_source_data`
- `failed`

OCR is intentionally selective. The pipeline marks `ocr_candidate` PDFs when text extraction is too sparse or produces no useful evidence sections. It does not run OCR unless an OCR engine is available in the environment.

## Current Benchmarks

Measured on 2026-05-04 with 50 live `Publicada` tenders from the PLACSP Bids search:

- Discovery fast path: 50/50 ready for Discovery in 5.675s wall time with 12 workers; p50 1.382s, p90 1.611s, max 1.748s per tender.
- Premium evidence path: 50/50 ready for Discovery and 28/50 `premium_ready` in 46.261s wall time with 6 workers; p50 4.409s, p90 7.767s, max 27.693s per tender.
- Remaining Premium blockers are mostly conservative guarantee extraction, confidence calibration where evidence exists but the score stays below threshold, and OCR/long-document exceptions.

Do not collapse these lanes. The product needs the fast path for near-real-time coverage and the premium path as asynchronous enrichment.

## Promote Premium Results Into The Prototype

After a live batch has premium packets, promote the recovered documents and source-backed detail into `discovery-v2` only after the full gate has passed:

```bash
node tools/simplifae-ingestion/promote_live_publicada_premium.js \
  --premium-root /tmp/simplifae-ingestion/quality-loop/01-.../premium \
  --quality-report /tmp/simplifae-ingestion/quality-loop/01-.../quality-gate.json \
  --semantic-report /tmp/simplifae-ingestion/quality-loop/01-.../semantic-audit.json \
  --parity-report /tmp/simplifae-ingestion/quality-loop/01-.../golden-parity.json \
  --packet-manifest /tmp/simplifae-ingestion/quality-loop/01-.../packet-manifest.json
```

The promotion step refuses to run if any required report is missing, has the wrong schema, belongs to a different premium root, or did not pass. It also promotes only packets marked `premium_ready` and `promotion_eligible` by the canonical manifest. Review/OCR/confidence packets stay visible only through the fast Discovery lane until they satisfy the premium evidence contract.

When it passes, the promotion step copies recovered PDFs into `discovery-v2/docs/live-publicada/`, generates `live-publicada-premium-overrides.js` for Tender Detail, and generates `live-publicada-viewer-docs.js` for the document viewer. This is the bridge between fast Discovery ingestion and premium source-backed Tender Detail.

If a tender has recoverable document URLs or premium packet documents, the validation job expects those documents to be promoted. Falling back to `Docs required` is valid only when documents are genuinely missing, unrecoverable, or still blocked by OCR/extraction gaps.

Render the promoted documents into the page-image contract used by `document-viewer.html`:

```bash
node tools/simplifae-ingestion/render_live_publicada_evidence_pages.mjs \
  --scale 1.35
```

By default this renders only the pages declared in each document's `availablePages` contract: page 1, source-backed evidence pages, and one page of immediate context around each evidence page. This is the public GitHub Pages package mode. It keeps source-chip/document-viewer navigation intact without committing huge PDFs or thousands of unused page images.

Use `--all-pages` only for local QA when the reviewer needs full in-document browsing. Full rendering is intentionally not the public publishing default because long project PDFs can exceed GitHub's file-size and repository-size limits.

Build and validate the public package explicitly:

```bash
node tools/simplifae-ingestion/build_live_publicada_public_package.mjs \
  --out-root /tmp/simplifae-live-publicada-public \
  --manifest-out /tmp/simplifae-ingestion/public-package-manifest.json \
  --pathspec-out /tmp/simplifae-ingestion/public-package-pathspec.txt

node tools/simplifae-ingestion/validate_tender_processing.js \
  --root /tmp/simplifae-live-publicada-public \
  --limit 50 \
  --runs 2 \
  --no-static \
  --out /tmp/simplifae-ingestion/public-package-validation.json
```

## Validation Job

Run this after changing scraper output, live tender data, or Tender Detail rendering:

```bash
NODE_PATH="/Users/victornavas/.cache/codex-runtimes/codex-primary-runtime/dependencies/node/node_modules" \
/Users/victornavas/.cache/codex-runtimes/codex-primary-runtime/dependencies/node/bin/node \
  tools/simplifae-ingestion/validate_tender_processing.js \
  --limit 50 \
  --runs 2 \
  --out /tmp/simplifae-ingestion/validation-report.json
```

The validator starts a local static server when `--url` is not provided, opens Discovery/Tender Detail in Playwright, and checks the live batch, static legacy tenders, and source-backed regression tenders. It reports:

- failing tender IDs and references
- failed section/component
- exception or console details
- input model snapshot that triggered the failure
- deterministic vs intermittent classification across repeated runs

It also enforces the Tender Detail evidence contract: source-backed detailed scope, required documents, and tender document rows must not contain fast-processing boilerplate, source chips must point to documents rendered in the prototype viewer, the viewer must not regress to a native PDF iframe, and evidence-rail chips must support cross-document navigation. Fast Discovery-only tenders should fall back to `Docs required` for premium sections until local viewer-backed evidence exists.

The job exits non-zero if any tender fails.

Use `--include-imported` for a heavier local sweep across all imported prototype opportunities in addition to the live/regression set.

Use `--no-static` only when intentionally isolating imported/live processing from the legacy static Tender Detail fixtures.

## Slow-Method Parity Quality Gate

Before promoting fast/premium output into the prototype, run the packet-level quality gate. It checks the recovered `premium.json` packets against the slow-method contract in `simplifae_slow_method_inventory.json`: source-backed document/page evidence, semantic separation of Detailed Scope, Required Documents, Admission Criteria and Award Criteria, no raw snippets as checklist rows, and no premium-ready status for OCR-blocked or evidence-missing tenders.

```bash
node tools/simplifae-ingestion/quality_gate.js \
  --premium-root /tmp/simplifae-ingestion/training-live-publicada-50-requirements-normalized-v2 \
  --out reports/simplifae-quality-gate-live-publicada-v2.json
```

The reusable mass-processing contract lives in `simplifae_field_contract.json`. This is the persistent version of what was learned from slow-method tenders: which fields Simplifae completes, which sources are allowed, which fields can be Discovery-fast, and which fields require document/page evidence before a tender can be considered premium-ready.

Run the golden parity benchmark when you want to compare a fast/premium batch against the current slow-method prototype references and the permanent field contract:

```bash
node tools/simplifae-ingestion/golden_parity_benchmark.js \
  --premium-root /tmp/simplifae-ingestion/quality-loop-20260505-fast-parity-v2/01-2026-05-05T21-24-42-456Z/premium \
  --out reports/simplifae-golden-parity-live-publicada-v2.json
```

Golden references are useful while we still have manually processed tenders, but they are not required for mass ingestion. For tenders with no slow reference, the same benchmark applies the field contract directly to each `premium.json`; missing slow snapshots are reported as `info` in browserless runs and as warnings only when browser snapshots are explicitly requested. Missing fast packets for golden references become warnings unless `--require-golden-packets` is explicitly set.

By default, the benchmark now runs browserless so it can execute in restricted CI/agent environments. Add `--browser-snapshots` when the environment can launch Playwright and you want to compare against live prototype data loaded from `discovery-v2/index.html`. Add `--require-browser-snapshots` only when browser snapshot failure should block the run.

The report includes:

- tender ID and reference
- failed section/component/step
- exception or rule details
- input packet snippet that triggered the failure
- deterministic classification for rule-based failures
- warnings for actionable quality gaps
- info signals for expected non-premium downgrades such as OCR candidates or unavailable browserless slow snapshots

The quality gate also emits `ocr_blockers`, separating `hard_ocr_blocker` documents with zero extracted text from `partial_text_review` documents where some text was recovered but premium certainty still needs OCR/legal review. This keeps mass ingestion fast while making OCR debt explicit and debuggable.

The same report includes `ocr_environment`, which records whether local OCR tooling is available (`tesseract` or `ocrmypdf`) and which PDF/OCR helper binaries are missing. This prevents OCR-related premium downgrades from becoming mystery failures in CI or on a local machine.

The OCR provider records per-document status in `premium.json`: `ocr_applied`, `ocr_provider`, `ocr_status`, `ocr_original_path`, `ocr_error`, and `ocr_elapsed_seconds`. The packet manifest surfaces provider warnings such as digitally signed PDFs that cannot be safely rewritten by `ocrmypdf`.

Build the canonical packet manifest after the quality gate when you need to verify document integrity and promotion eligibility explicitly:

```bash
node tools/simplifae-ingestion/packet_manifest.js \
  --premium-root /tmp/simplifae-ingestion/quality-loop/01-.../premium \
  --quality-report /tmp/simplifae-ingestion/quality-loop/01-.../quality-gate.json \
  --out /tmp/simplifae-ingestion/quality-loop/01-.../packet-manifest.json
```

The manifest is now part of `run_quality_loop.js` and `simplifae.js smoke/full-50`. It catches packet-level integrity regressions that structural extraction can miss, such as source chips pointing to a page outside the recovered PDF.

Use the loop runner when iterating on the scraper. It chains premium extraction, quality gate, promotion, page rendering and UI validation, stopping at the first failing layer with a report artifact:

```bash
node tools/simplifae-ingestion/run_quality_loop.js \
  --seed-file /tmp/simplifae-ingestion/live-publicada-50.json \
  --limit 50 \
  --workers 6 \
  --semantic-audit \
  --parity-benchmark \
  --max-iterations 1 \
  --out-root /tmp/simplifae-ingestion/quality-loop
```

`--semantic-audit` is stricter than the structural quality gate. It catches premium packets whose shape is valid but whose content is not yet slow-method quality: generic requirement rows, weak evidence snippets, index-page evidence, missing award weights, CPV pollution, and evidence/category mismatches.


After changing the structural quality gate or OCR observability, run:


After changing the command wrapper, run:

As of 2026-05-06, the representative 50-tender live `Publicada` batch passes with `--semantic-audit --parity-benchmark` and canonical manifest validation: 50/50 processed, 0 structural failures, 0 semantic failures, 0 semantic warnings, 0 parity failures, and 0 manifest failures. The same dry-run promotion promotes 19 premium-ready tenders and skips 31 review/OCR/confidence tenders instead of publishing incomplete premium certainty.

With local OCR enabled on the same 50-tender batch: 50/50 processed, 20 `promotion_eligible`, 30 review/OCR/confidence tenders, 30 OCR applications, 0 hard OCR blockers, and 3 OCR provider warnings for documents that could not be safely rewritten.

For a fast smoke test against an existing premium batch without mutating the prototype:

```bash
node tools/simplifae-ingestion/run_quality_loop.js \
  --skip-train \
  --premium-root /tmp/simplifae-ingestion/training-live-publicada-50-requirements-normalized-v2 \
  --skip-promote \
  --skip-render \
  --skip-ui-validation \
  --out-root /tmp/simplifae-ingestion/quality-loop-smoke
```

This is the intended loop for silent refinement: make one scraper/classification change, rerun the loop, inspect the first failing layer, fix the root-cause class, and repeat until both packet quality and UI validation pass for the representative batch. Correctness wins over speed; if a tender cannot satisfy the slow-method evidence contract, it must be downgraded to review/OCR/docs-required instead of being promoted as premium-ready.

## Current Design Constraint

The script does not mutate `discovery-v2/index.html` or `discovery-v2/document-viewer.html`. It is the staging layer we need before generating a premium Tender Detail automatically.
